Socket Programming Survival Guide

This guide is for students building the Project 1 peer-to-peer file-sharing system. It does not hand you a canned solution. It gives you the engineering shape of a strong solution: how to structure the system, how to think about sockets, what state to keep, where concurrency matters, and where students usually lose points.

What the instructor actually wants

1. A real overlay

Not a simulation inside one process. You are expected to run one node per machine on the designated dc machines and connect them with stream sockets.

2. Concurrency

The network creation protocol should not be lazily sequential. A design that creates many links in parallel will score better than one that walks the graph slowly.

3. Protocol design

You must explain your message formats, module boundaries, routing behavior, and cleanup strategy in the design document.

4. Engineering discipline

You are allowed to use AI, but you are still responsible for understanding every line and cleanly separating human-written, AI-assisted, and AI-generated code.

Important: the assignment is not just about file transfer. It is equally about distributed protocol design, state management, correct forwarding, and clean demonstrations.

A sane system model

A

Overlay layer

Maintains TCP neighbor connections implied by Adj, handles join/leave updates, and provides send/receive primitives to upper layers.

B

Search layer

Creates unique search requests, enforces hop-count rules, forwards each request at most once, and routes replies back along the reverse path.

C

File layer

Checks the local share list F_i, serves file downloads, updates local file visibility after successful copies, and blocks unsafe departure.

Keep overlay logic separate from search logic Treat file transfer as a separate protocol Log everything that affects correctness Expire stale search state

Part-by-part roadmap

Part I — Build the graph first

  • Parse the global adjacency matrix and node-to-machine registry.
  • Verify the graph is connected before trying to build anything.
  • Use a deterministic connection rule to avoid duplicate TCP links.
  • Bring up the full overlay concurrently, not one edge at a time.

Part II — Get search correct before optimizing

  • Define a unique request ID, for example (originNode, sequenceNumber).
  • Store the first neighbor that delivered each request.
  • Forward each search request at most once.
  • Ignore late replies after the initiator timer expires.

Part III — Join and leave without chaos

  • Update topology state in a clearly documented way.
  • Wait for in-flight transfers to finish before departure.
  • Refuse new positive search replies while departing.
  • If attempting the bonus, reason carefully about partitions.

How to use the UTD dc machines without getting stuck

The assignment expects you to use the designated network-programming machines such as dc01.utdallas.edu through dc45.utdallas.edu, with one node per machine. For many students, this is the first real multi-machine workflow, so the safest approach is to set up access first, then worry about the distributed code.

Step 1 — Get SSH working

If you are on macOS, Linux, or WSL, generate an SSH key with:

ssh-keygen -t ed25519

Accept the default file location unless you know exactly why you want something else. This creates:

  • ~/.ssh/id_ed25519 — your private key
  • ~/.ssh/id_ed25519.pub — your public key
Critical: share only the public key. Never copy, upload, or send the private key anywhere.

Step 2 — Install the public key on a dc machine

The easiest method is usually:

ssh-copy-id -i ~/.ssh/id_ed25519.pub NetID@dc01.utdallas.edu

If that is unavailable, you can log in normally first and manually append the contents of your public key file on the remote side.

After installation, test with ssh NetID@dc01.utdallas.edu. You may be asked for your key passphrase, which is normal.

Step 3 — Build a machine map before coding

Create a small config file that maps logical nodes to actual hosts and ports, for example:

Node1  dc01.utdallas.edu  5001
Node2  dc02.utdallas.edu  5002
Node3  dc03.utdallas.edu  5003
Node4  dc04.utdallas.edu  5004

This makes your registry explicit and keeps your design document cleaner.

Step 4 — Keep one terminal per node

For debugging, act like a distributed-systems engineer, not like someone hoping it will all work magically. Use one shell per remote node, or use tmux so each machine keeps running even if your laptop disconnects.

Step 5 — Compile and test remotely

Do not assume code that worked on your local machine behaves the same way across multiple remote hosts. Compile on the dc machines, verify ports, and test on at least a 3–4 node overlay early.

Step 6 — Log what runs where

Write down which process ran on which machine, with which node ID, port, and visible file list. This prevents confusion during the demo and when debugging reply routing.

Recommended student workflow: first verify SSH access to several dc machines, then test a tiny two-node hello program, then test one overlay edge, then a small connected graph, and only after that attempt flooding and file transfer.

Socket programming essentials for this project

The TCP pattern you need to internalize

Passive side (server-ish)          Active side (client-ish)
-------------------------------------------------------------
socket()                           socket()
bind()                             connect()
listen()                           send()/recv()
accept()                           close()
send()/recv()
close()

Every node will likely play both roles: it must accept neighbor/file-transfer connections and also initiate connections to neighbors or download peers.

Practical recommendation: use one well-defined listening port per node for overlay control traffic, and either reuse the same connection carefully or create a separate, explicit transfer connection for file downloads.

Three socket mistakes that destroy distributed projects

  • Assuming one send() equals one recv(). TCP is a byte stream, not a message queue.
  • Ignoring partial reads/writes. You must frame your messages.
  • Blocking the whole node on one connection. Use threads or select()/poll() so one slow peer does not freeze everything.

How to actually write the socket code

Students usually lose time here because they jump straight into the full P2P protocol before writing a tiny, correct socket layer. The right move is to build a small communication library first, then let the rest of the system use it.

Start with these helper functions

int send_all(int fd, const void *buf, size_t len);
int recv_all(int fd, void *buf, size_t len);
int send_message(int fd, uint16_t type, const void *payload, uint32_t len);
int recv_message(int fd, uint16_t *type, void **payload, uint32_t *len);

If these helpers are correct, your search, reply, join, leave, and download code becomes much simpler.

Use a fixed header

A tiny header keeps control messages predictable.

typedef struct {
    uint16_t type;
    uint32_t length;
} MsgHeader;

Send the header first, then the payload. Use network byte order for multi-byte integers.

Server-side skeleton

listen_fd = socket(...);
bind(listen_fd, ...);
listen(listen_fd, backlog);

while (1) {
    conn_fd = accept(listen_fd, ...);
    // hand off to a thread or event loop
}

Client-side skeleton

fd = socket(...);
connect(fd, ...);
send_message(fd, TYPE_SEARCH_REQ, payload, len);
recv_message(fd, &type, &payload, &len);
close(fd);

Overlay reality

Each node is both a server and a client. It must accept inbound neighbor connections and initiate outbound ones according to your connection rule.

Transfer reality

File transfer is safer if treated as its own mini-protocol with metadata first, then exact byte counts, then closure and cleanup.

What send_all() must do

  • Loop until every byte is sent or an error occurs.
  • Handle short writes correctly.
  • Return failure if the socket breaks mid-message.

What recv_all() must do

  • Loop until the requested byte count is fully read.
  • Detect peer closure cleanly.
  • Never assume one recv() call returns a whole header or payload.
The classic mistake: writing code like “call recv() once for the header, call it once for the payload, done.” That works until it doesn’t. TCP gives you a stream of bytes, not packet boundaries.

Concurrency choice

At minimum, each node needs to handle multiple neighbors without letting one stalled connection freeze the process. Threads are usually easier for students; select() or poll() is also valid if done cleanly.

Error handling

Check every system call. If socket(), bind(), listen(), accept(), connect(), send(), or recv() fails, log the failure and decide whether to retry, abort that connection, or terminate the node.

Byte order

If your header contains integers, convert them with the usual host/network byte-order helpers so your protocol is explicit and portable.

Close sockets on all paths

Leaking file descriptors during failures or repeated searches will create weird bugs that look like distributed-systems failures but are really resource leaks.

Recommended implementation order: first make a two-process echo or hello program work across two dc machines, then add a header, then add message types, then build the overlay on top of that. Do not start with flooding.

Use explicit message framing

Since you are using stream sockets, you need a message format that can be parsed unambiguously. Two clean approaches are common:

Option 1 — Length-prefixed messages

Send a fixed-size header first, containing the message type and payload length, then send the payload bytes.

Best for binary-safe protocols and file chunks.

Option 2 — Newline-delimited JSON

Each control message is one JSON object terminated by \n. Easier to debug, but you still need length framing for actual file chunks.

Great for search, reply, join, and leave control traffic.

What not to do

Do not invent a format that only works when messages arrive in one piece. Under real load, that assumption breaks fast.

Strong compromise

Use structured control messages for overlay/search management and a separate framed transfer protocol for file data.

Recommended control messages

Message type Purpose Suggested fields
HELLO Neighbor handshake during overlay creation nodeId, host, port, protocolVersion
SEARCH_REQ Flooded file search request requestId, originNode, filename, hopLimit, hopDistance, timestamp
SEARCH_RESP Reply when a node has the file requestId, ownerNode, filename, ownerHost, ownerPort
DOWNLOAD_REQ Initiate actual file transfer filename, requesterNode, optional byte range
FILE_META Transfer metadata before file bytes filename, filesize, checksum, rangeStart, rangeEnd
JOIN_REQ New node wants to enter through an existing node newNodeId, host, port, visibleFiles
TOPOLOGY_UPDATE Propagate adjacency/registy changes changeType, nodeA, nodeB, updatedVersion
LEAVE_NOTICE Graceful departure departingNode, effectiveTime, replacementEdges(optional)
ACK / ERROR Protocol confirmation and debugging requestId, status, reason
Do not skip the request ID. It is the anchor for duplicate suppression, reverse-path routing, timeout cleanup, and debugging.

A design that earns points for concurrency

Deterministic connection initiation

If Adj[i][j] = 1, only one side should actively connect. A simple rule is:

If i < j, Node i initiates TCP connect to Node j.
If i > j, Node i only accepts the incoming connection.

This avoids duplicate sockets while still allowing all edges to come up in parallel.

Parallel startup

Do not wait for one connection to finish before starting the next. Launch connection attempts to all eligible neighbors concurrently, and separately accept inbound connections.

Why this matters: the project statement explicitly says a high-concurrency protocol will fetch more points, while a sequential setup will be penalized.

The search protocol that will actually work

  1. User searches at Node i. If the filename is already in F_i, serve locally.
  2. Otherwise create a fresh request ID. Example: requestId = i || localSequenceNumber.
  3. Initialize hop-count to 1. Flood the request only within that search radius.
  4. Each node stores first arrival only. If a duplicate request arrives later, discard it.
  5. Nodes with a matching file reply along the reverse path. They respond to the neighbor from which the first copy arrived.
  6. Intermediate nodes forward replies using stored reverse-path state.
  7. The initiator collects replies until timeout. Display (filename, node) tuples and ignore late arrivals.
  8. If no success, double hop-count. Retry until success or hop-count exceeds 16.
Search state per request at each node
-------------------------------------
requestId
firstSeenTime
parentNeighbor   # the first neighbor from which request arrived
alreadyForwarded # boolean
cleanupDeadline  # expire stale routing state
Easy place to lose correctness: if you fail to remember the first incoming neighbor for a request, replies will not reliably find their way back to the initiator.

Data structures worth having

Structure What it stores Why it matters
PeerRegistry nodeId → host, port, machine name Lets any node know how to contact others and supports join/leave bookkeeping.
AdjacencyState Current neighbor graph or local neighbor list Used for connection creation, flooding, and topology updates.
VisibleFiles The contents of F_i Defines what the node may share and what counts as local success.
SeenRequests Set or map keyed by request ID Enforces “forward at most once” and prevents flooding loops.
ReversePathTable request ID → parent neighbor Routes replies back to the search originator.
InFlightTransfers Current uploads/downloads Needed for graceful departure and transfer cleanup.

Join and departure design hints

Join

  • New node contacts a known existing node.
  • Both sides update the registry and adjacency state.
  • Establish the new neighbor socket immediately.
  • Make sure the rest of the overlay eventually learns the update.

Graceful leave

  • Mark the node as departing.
  • Stop answering file searches positively.
  • Wait until all active transfers finish.
  • Then close neighbor sockets and propagate topology updates.

Bonus leave logic

  • Estimate whether your departure partitions the graph.
  • Pick one neighbor from each would-be component.
  • Ask them to form a ring before you leave.
  • Document the algorithm carefully if you attempt it.
Minimal but solid strategy: make topology updates versioned and idempotent. If nodes receive the same update twice, they should not break.

Likely failure modes

Duplicate TCP links

Happens when both endpoints attempt to connect to each other at the same time without a deterministic rule.

Infinite flooding

Happens when request IDs are not unique or duplicates are not suppressed immediately.

Broken reply routing

Happens when a node does not remember the first incoming neighbor for a request.

Stale search state

Happens when request records never expire. The assignment explicitly says not to keep this state around for too long.

Bad message parsing

Happens when the implementation assumes TCP preserves message boundaries.

Unsafe node departure

Happens when a node leaves during an upload/download or still answers new searches while shutting down.

A strong design document structure

  1. System overview. What processes run, where they run, and how they are identified.
  2. Module layout. Separate connection management, search, file transfer, topology state, CLI, and logging.
  3. Data structures. Explain every shared map/table and when entries are inserted or removed.
  4. Message protocol. Define each message type and every field it carries.
  5. Initial overlay creation. Explain the connectivity check and concurrent connection protocol.
  6. Search and reply routing. Explain duplicate suppression, hop-count logic, timers, and late-reply behavior.
  7. File transfer. Explain download setup, transfer termination, and F_i updates.
  8. Join and leave. Explain topology updates, graceful departure, and cleanup.
  9. Performance logging. Explain how you record chosen hop-count and reply latency.
  10. AI-use disclosure. Separate human-written, AI-assisted, and AI-generated code honestly.

Final checklist before you submit or demo

Correctness

  • One node per machine.
  • No duplicate forwarding of search requests.
  • Reply routing works from matches back to initiator.
  • Late replies are ignored.
  • Graceful leave waits for transfers to finish.

Presentation

  • Your design document names modules and message formats clearly.
  • Your demo shows overlay creation, a successful search, and a file transfer.
  • Your logs show the chosen hop-count and reply latencies.
  • Your AI-use statement is complete and honest.

Bottom line: the best submissions will not look like giant monolithic programs. They will look like a clean distributed system: deliberate message types, bounded state, clear concurrency choices, and evidence that the student understands why the protocol works.
